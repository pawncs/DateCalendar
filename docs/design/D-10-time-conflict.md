# D-10: 时间冲突检测

## 1. 必要性 (Why)

### 问题
当用户在同一时间段安排了多个日程，系统应该给出提醒。否则用户可能在不知情的情况下 double-book，导致计划无法执行。

### 场景
- 用户在 14:00-15:00 创建了新日程 → 系统提示「该时段已有『项目评审』(14:00-16:00)」
- 用户在周视图中看到周三下午排了 3 个会 → 系统用红色标记冲突时段
- 用户拖拽调整日程时间 → 系统实时检测并提示冲突

### 冲突类型
- **完全重叠**：A(14:00-15:00) 与 B(14:00-15:00)
- **部分重叠**：A(14:00-16:00) 与 B(15:00-17:00)
- **包含关系**：A(14:00-17:00) 包含 B(15:00-16:00)

---

## 2. 实现方案 (How)

### 2.1 冲突检测算法

```
两个时间段 [start1, end1) 和 [start2, end2) 冲突 ⇔ start1 < end2 AND start2 < end1
```

前端工具函数 `hasTimeConflict` 和 `findConflicts`。

### 2.2 后端冲突查询

`check_conflicts` 方法：查询与指定时间段重叠的 fixed 类型日程（排除已取消和自身）。

### 2.3 前端冲突展示

**创建日程时的冲突提醒**：ScheduleEditor 中实时检测（debounce 300ms），冲突时显示黄色警告框，列出冲突日程。

**日/周视图中的冲突标记**：冲突的日程块用红色边框标记。冲突日程并排显示（各占 50% 宽度）。

### 2.4 冲突检测时机

| 时机 | 检测方式 | UI 反馈 |
|------|----------|---------|
| 创建日程时 | 实时检测（debounce 300ms） | 表单底部冲突列表 |
| 编辑日程时 | 同上 | 同上 |
| 日/周视图加载时 | 加载完所有日程后检测 | 红色边框 + 并排显示 |

## 3. 验证标准 (Verify)

### 功能验证

| 测试场景 | 预期结果 | 验证方法 |
|----------|----------|----------|
| 完全重叠检测 | 创建 14:00-15:00，已有同时段日程 → 显示冲突 | 手动操作 |
| 部分重叠检测 | 创建 14:30-15:30，已有 14:00-15:00 → 冲突 | 手动操作 |
| 不冲突 | 创建 15:00-16:00，已有 14:00-15:00 → 无冲突 | 手动操作 |
| 编辑排除自身 | 编辑已有日程时不将自身标记为冲突 | 手动操作 |
| 视图红色标记 | 冲突日程在日/周视图中显示红色边框 | 目视确认 |
| 并排显示 | 两个冲突日程各占 50% 宽度 | 目视确认 |
| 已取消日程不检测 | cancelled 的日程不参与冲突检测 | 手动操作 |
| 待办不检测 | todo_day/todo_week 类型不参与冲突检测 | 手动操作 |

### 边界条件

| 场景 | 预期 |
|------|------|
| 无冲突 | 不显示任何警告 |
| 3 个以上冲突 | 全部列出，各占 1/3 宽度 |
| 跨天冲突 | 只检测同一天的冲突 |

### 交互体验验证

| 场景 | 预期 |
|------|------|
| 冲突提示不阻塞 | 冲突是警告，用户可以忽略并保存 |
| 实时响应 | 修改时间后 300ms 内显示冲突 |

# D-03: 批量操作

## 1. 必要性 (Why)

### 问题
当前任务操作（完成、删除、移动）都是逐条进行的。当用户完成一个里程碑后想批量完成其下所有子任务，或想清理一批已取消的任务时，逐条操作效率极低。

### 场景
- 里程碑收尾：完成一个里程碑 → 一键批量完成其下所有子任务
- 大扫除：筛选所有 cancelled 任务 → 批量删除
- 任务重组：选中多个任务 → 批量移动到另一个里程碑下

### 操作范围
Phase 2 实现三种批量操作：**批量完成**、**批量删除**、**批量移动**。

---

## 2. 实现方案 (How)

### 2.1 整体流程

```
用户进入选择模式（长按/点击复选框图标）
  → 勾选多个任务节点
  → 底部出现批量操作工具栏
  → 选择操作（完成/删除/移动）
  → 确认弹窗（删除需确认）
  → 调用后端批量接口
  → 刷新任务树
```

### 2.2 前端实现

**taskStore 扩展**：新增 `selectionMode`、`selectedIds`、`enterSelectionMode`、`exitSelectionMode`、`toggleSelection`、`batchComplete`、`batchDelete`、`batchMove`。

**TaskNode 改造**：每个节点左侧添加复选框，仅在 `selectionMode` 为 true 时显示。

**新建批量操作工具栏**：`src/components/tasks/BatchActionBar.tsx`，固定在底部浮层，显示选中数量 + 批量完成/删除/移动按钮 + 取消按钮。

**确认弹窗**：批量删除需 AlertDialog 二次确认（不可逆操作）。

### 2.3 后端实现

**批量完成**：`batch_update_status` — 一条 SQL 更新多个任务的状态（`WHERE id IN (...)`）。

**批量删除**：`batch_delete` — 循环调用已有的递归删除逻辑（含子任务、日程、笔记、风险的级联删除）。

**批量移动**：`batch_move` — 计算目标父节点下的最大 sort_order，依次分配新位置。

**Tauri 命令**：`batch_update_tasks`、`batch_delete_tasks`、`batch_move_tasks`。

## 3. 验证标准 (Verify)

### 功能验证

| 测试场景 | 预期结果 | 验证方法 |
|----------|----------|----------|
| 选择单个任务 | 点复选框 → 该任务被选中，工具栏出现 | 手动操作 |
| 选择多个任务 | 勾选 3 个 → 工具栏显示「已选 3 项」 | 手动操作 |
| 批量完成 | 选 3 个 pending → 点批量完成 → 全部变 completed | 手动 + 数据库查询 |
| 批量删除 | 选 2 个 → 确认弹窗 → 确认 → 任务及子任务消失 | 手动 + 数据库查询 |
| 批量移动 | 选 2 个 → 批量移动 → 选目标父节点 → 移动成功 | 手动操作 |
| 取消选择 | 点取消 → 退出选择模式，复选框消失 | 手动操作 |
| 删除确认 | 点批量删除 → 弹窗 → 点取消 → 不执行删除 | 手动操作 |
| 子任务级联删除 | 删除父任务 → 子任务一并删除 | 数据库查询 |

### 边界条件

| 场景 | 预期 |
|------|------|
| 选择 0 个任务时点批量操作 | 按钮置灰，不可点击 |
| 选择全部任务后批量删除 | 确认弹窗显示正确数量 |
| 批量移动时目标为自己 | 阻止操作，不产生循环引用 |

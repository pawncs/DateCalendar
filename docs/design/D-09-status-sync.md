# D-09: 日程状态同步

## 1. 必要性 (Why)

### 问题
当前 Task 和 Schedule 是两个独立的实体，各有各的 `status` 字段。但实际使用中，它们的状态应该联动：完成日程 → 关联任务也应完成；完成任务 → 关联日程也应完成。如果没有同步机制，用户需要两处分别操作，体验割裂。

### 场景
- 用户在待办列表中打钩「代码审查」→ 任务树中「代码审查」任务自动变为 completed
- 用户完成任务树中的「项目启动」里程碑 → 关联的所有日程自动完成
- 用户取消完成一个任务 → 关联日程恢复为 pending

---

## 2. 实现方案 (How)

### 2.1 同步规则

| 触发操作 | 联动操作 |
|----------|----------|
| 日程 → completed | 关联任务 → completed |
| 日程 → pending | 关联任务 → in_progress |
| 任务 → completed | 关联日程 → completed |
| 任务 → cancelled | 关联日程 → cancelled |
| 任务删除 | 关联日程级联删除 |

### 2.2 后端实现

**schedule_service 新增 `update_schedule_status`**：
1. 更新日程状态
2. 查询关联 task_id
3. 同步更新任务状态（completed ↔ in_progress）
4. 全部在一个事务中完成

**task_service 新增 `update_task_status`**：
1. 更新任务状态
2. 同步更新关联的所有日程状态
3. 全部在一个事务中完成

🔍 知识点雷达: 数据库事务
   ├── 是什么: 一组操作要么全部成功，要么全部回滚，保证数据一致性
   ├── 为什么用: 状态同步涉及两张表（tasks + schedules），必须原子执行
   ├── 核心心智模型: BEGIN → 操作1 → 操作2 → COMMIT（成功）/ ROLLBACK（失败）
   └── 关联概念: SQLite 默认自动提交，手动事务可以保证多表操作的原子性

### 2.3 级联删除增强

在 `delete_task_recursive` 中增加删除关联日程的步骤（`DELETE FROM schedules WHERE task_id = ?`）。

### 2.4 避免循环同步

通过 `WHERE status != 'completed'` 条件避免重复更新。日程完成 → 任务完成（条件：任务还不是 completed）→ 不再触发日程更新。

### 2.5 前端感知

前端不需要额外处理同步逻辑——后端在同一个事务中完成所有更新，前端只需要在操作后重新加载数据（`loadTasks` + `loadDaySchedules`）。

## 3. 验证标准 (Verify)

### 功能验证

| 测试场景 | 预期结果 | 验证方法 |
|----------|----------|----------|
| 日程完成 → 任务完成 | 待办列表打钩 → 任务树中关联任务变为 completed | 手动操作 → 检查两处 |
| 任务完成 → 日程完成 | 任务树中完成任务 → 关联日程全部变为 completed | 手动操作 |
| 日程取消完成 → 任务恢复 | 取消打钩 → 任务恢复为 in_progress | 手动操作 |
| 任务取消 → 日程取消 | 任务标记为 cancelled → 关联日程全部 cancelled | 手动操作 |
| 删除任务 → 级联删除日程 | 删除任务 → 关联日程消失 | 手动操作 + 数据库查询 |
| 避免循环 | 快速连续操作 → 不出现死循环或重复更新 | 多次快速操作 |
| 事务原子性 | 模拟错误 → 两张表都不变化 | 代码审查事务逻辑 |

### 边界条件

| 场景 | 预期 |
|------|------|
| 日程无关联任务 | 只更新日程状态，不报错 |
| 任务已完成再完成 | 不重复更新 |
| 多个日程关联同一任务 | 完成其中一个 → 任务也完成；其他日程也变为完成 |
